### 이번주 스터디 목표

### 오늘의 키워드

**1. Repository Layer**
- 데이터베이스에 접근하여 CRUD 작업을 수행하는 계층
- Create : 데이터 생성
- Read : 데이터 조회
- Update: 기존 데이터 수정
- Delete : 데이터 삭제

**2. ORM과 JPA**
- 데이터베이스에 접근 -> 애플리케이션 서버는 DB가 이해할 수 있는 SQL을 연결된 커넥션으로 DB에 전달 -> DB는 전달된 SQL을 수행하고 그 결과를 응답, 애플리케이션 서버는 결과 활용
- SQL 직접 다루는 방식: 반복적인 SQL 코드 작성, SQL에 의존적인 개발, 객체 - 데이터베이스 간 패러다임 불일치 => ORM
- ORM: 객체와 관계형 데이터베이스의 데이터를 자동으로 매핑
- JPA: 자바 진영의 ORM 기술 표준, 자바에서 객체를 데이터베이스에 저장하고 관리하기 위한 인터페이스와 기능을 제공하는 API

**3. JPA 동작 방식**
- EntityManager: 엔티티 객체를 데이터베이스와 연결해주는 창구 역할
- 엔티티 매니저가 대부분 기능 제공
- 엔티티 매니저를 사용해 SQL을 직접 작성하지 않고도 엔티티를 데이터베이스에 등록/수정/조회/삭제 가능
- 영속성 컨텍스트: JPA에서 엔티티를 관리하기 위해서 만들어져있는 가상 공간, 엔티티 매니저를 통해 영속성 컨텍스트에 접근
- 1차 캐시, 동일성 보장, 트랜잭션을 지원하는 쓰기 지연, 변경 감지
- 트랜잭션: 데이터베이스에서 수행되는 작업의 단위, 여러개의 데이터베이스 연산을 하나의 논리적인 작업 단위로 묶어서 실행
- 1차 캐시: 한 번 조회한 객체를 영속성 컨텍스트가 내부에 저장해두고 재사용하는 구조
- 동일성 보장: 트랜잭션 내에서는 같은 엔티티를 여러번 조회해도 영속성 컨텍스트에 보관된 같은 객체를 그대로 반환 -> 항상 동일한 객체 보장
- 변경 감지: 영속성 컨텍스트가 엔티티의 변경 사항을 자동으로 감지, 트랜잭션 커밋 시점에 변경 내용을 DB에 반영하는 것, 영속성 컨텍스트가 관리하는 영속 상태의 엔티티에만 적용

### 느낀점
이번 주는 기본기를 다시 다지는 복습 중심의 시간이었고, ORM과 JPA의 구조적 흐름을 이해하는 데 큰 도움이 되었다. 특히 엔티티 매니저와 영속성 컨텍스트의 관계를 통해 JPA가 SQL을 자동으로 처리하는 원리를 명확히 알 수 있었다. 단순 CRUD가 아니라 트랜잭션과 변경 감지까지 포함된 데이터 흐름 전체를 이해하게 되어 실무 감각이 한층 올라갔다.
